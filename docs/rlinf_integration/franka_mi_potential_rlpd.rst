#############################################################
MI Potential Shaping Reward on Franka
#############################################################

.. note::

   This document follows the structure of the `RLinf Real-World RL docs <https://rlinf.readthedocs.io/zh-cn/latest/rst_source/examples/embodied/franka.html>`_
   and `Reward Model guide <https://rlinf.readthedocs.io/zh-cn/latest/rst_source/extending/reward_model.html>`_.
   The MI potential reward model is the fourth type alongside RLinf's existing ``resnet`` / ``vlm`` / ``history_vlm`` reward models.

----

Overview
========

MI Potential Shaping is an **offline distillation, online shaping** reward scheme:

- **Offline phase**: A lightweight StatePotentialRewardModel :math:`V_\phi(o_t, g)` is distilled from world-model-generated futures via Dame-style temporally-aligned mutual information scoring.
- **Online phase**: The frozen :math:`V_\phi` computes the potential difference :math:`\gamma V_\phi(o_{t+1}) - V_\phi(o_t)` at each RLPD training step, producing a dense "progress" shaping signal added on top of the sparse environment reward.

**Online inference does NOT run** the world model, B-spline MI, temporal alignment, or pseudo-preference generation. Only the frozen student model runs.

Comparison with RLinf's existing reward models:

+---------------------+---------------------+---------------------+------------------------------+
| Property             | ResNet / VLM        | HistoryVLM          | **MI Potential (this work)**   |
+---------------------+---------------------+---------------------+------------------------------+
| Output semantics     | Success prob (0/1)  | Success prob (0/1)  | **State potential (scalar)**    |
| Reward computation   | Model output direct  | Model output direct  | **Potential diff**: γV(next)-V(curr) |
| Training data        | Human labels         | Human labels         | **Auto-generated by world model** |
| Needs real robot data| Yes                 | Yes                 | **No for pseudo-labels; yes for validation** |
| Online latency       | Low                 | Medium (multi-frame VLM) | **Very low (GRU, ~2M params)** |
| Model registry type  | resnet / vlm        | history_vlm         | **mi_potential**              |
+---------------------+---------------------+---------------------+------------------------------+

.. warning::

   This method is still experimental. Never rely solely on MI potential signals
   to control the robot in safety-critical scenarios. MI potential **cannot**
   trigger episode termination, determine task success, or bypass collision /
   force / workspace limits.

Prerequisites
=============

1. The RLinf Franka base deployment is complete; ``run_realworld_async.sh`` starts successfully.
2. Camera calibration and network configuration are done per ``franka.rst``.
3. **Offline MI reward model training is complete**, producing:

   .. code-block:: text

      mi_student_checkpoint/
      ├── pytorch_model.pt      # auto-generated by training script
      └── metadata.json         # manually or script-generated (see below)

4. The ``MI-directional-WAM-reward-model-pretrain`` repository is importable on the training node:

   .. code-block:: bash

      cd /path/to/MI-directional-WAM-reward-model-pretrain
      pip install -e .

      python -c "from mi_reward.inference.potential_model import MIPotentialInferenceModel; print('OK')"

Step 1: Export the MI Student Checkpoint
=========================================

After offline training completes, generate ``metadata.json`` in the
``MI-directional-WAM-reward-model-pretrain`` repository:

.. code-block:: bash

   cd /path/to/MI-directional-WAM-reward-model-pretrain

   python - << 'PY'
   import json
   from pathlib import Path
   import torch

   ckpt_path = "results/mi_reward/reward_head_mvp/pytorch_model.pt"
   ckpt = torch.load(ckpt_path, map_location="cpu")
   config = ckpt.get("config", {})

   metadata = {
       "checkpoint_format_version": "1.0",
       "model_class": "StatePotentialRewardModel",
       "encoder_name": "dino_v3",
       "input_image_keys": ["main_images"],
       "history_size": 1,
       "task_conditioned": False,
       "output_semantics": "state_potential",
       "training_gamma": 0.99,
       "architecture": "gru",
       "token_pooling": "mean",
       "num_layers": 2,
       "num_heads": 4,
       "hidden_dim": config.get("hidden_dim", 256),
       "input_dim": config.get("input_dim", 512),
       "potential_mean": 0.0,
       "potential_std": 1.0
   }

   out = Path(ckpt_path).parent / "metadata.json"
   out.write_text(json.dumps(metadata, indent=2, ensure_ascii=False))
   print(f"Written to {out}")
   PY

Copy the checkpoint directory to the training node:

.. code-block:: bash

   scp -r results/mi_reward/reward_head_mvp/ user@gpu-node:/path/to/checkpoints/mi_potential/

Step 2: Configure YAML
========================

Starting from the existing Franka RLPD config, create
``realworld_peginsertion_rlpd_cnn_async_mi_potential.yaml``:

.. code-block:: yaml
   :caption: examples/embodiment/config/realworld_peginsertion_rlpd_cnn_async_mi_potential.yaml

   # Inherit the standard Franka RLPD config
   defaults:
     - realworld_peginsertion_rlpd_cnn_async
     - _self_

   # =====================================================================
   # Reward: enable MI potential shaping
   # =====================================================================
   reward:

     # --- Basic settings ---
     use_reward_model: true
     group_name: "RewardGroup"

     # Must be false: reward model runs on the GPU training node
     standalone_realworld: false

     # Per-step reward computation
     reward_mode: "per_step"

     # Declare: model returns state potential, not binary success prob
     reward_output_type: "state_potential"

     # Environment reward weight — keep sparse task reward
     env_reward_weight: 1.0

     # MI potential reward weight — tune shaping strength
     reward_weight: 1.0

     # ===================================================================
     # Potential shaping parameters
     # ===================================================================
     potential_shaping:

       enabled: true

       # Potential decay factor, default matches SAC gamma
       gamma: ${algorithm.gamma}

       # Potential-difference scale factor
       scale: 1.0

       # Clip potential difference to [-delta_clip, +delta_clip]
       delta_clip: 1.0

       # Confidence gating
       confidence_gate: true

       # Confidence combination: "min" or "geometric_mean"
       confidence_combine: "min"

       # Minimum confidence threshold
       min_confidence: 0.0

       # Invalid prediction handling: "zero_shaping" (skip shaping, keep env reward)
       invalid_behavior: "zero_shaping"

       # Potential normalization (recommend off for initial runs)
       normalize: false

       # Warmup steps (no shaping for first N steps)
       warmup_steps: 0

       # [Safety] Potential must not be used for termination or success
       use_for_termination: false
       use_for_success: false

     # ===================================================================
     # MI potential model config
     # ===================================================================
     model:
       # Registered as mi_potential type
       model_type: "mi_potential"

       # Path to the trained student checkpoint
       model_path: /path/to/checkpoints/mi_potential/pytorch_model.pt

       # Optional: training YAML config
       config_path: null

       # Inference precision
       precision: "fp32"
       device: "cuda"

       # Input mode: "current_frame" (single) or "history_buffer" (multi-frame window)
       input_mode: "current_frame"

       # Which camera images to use
       image_keys:
         - main_images

       # Task description (current model does not use task conditioning; placeholder)
       task_description: "insert the peg into the target hole"

   # =====================================================================
   # Phase B: MI-guided replay sampling (optional, disabled by default)
   # =====================================================================
   algorithm:
     mi_guided_replay:
       enabled: false
       apply_to: "online_only"
       positive_threshold: 0.02
       negative_threshold: -0.02
       min_confidence: 0.2
       positive_fraction: 0.4
       neutral_fraction: 0.2
       negative_fraction: 0.4
       fallback_to_uniform: true
       max_resample_attempts: 5

   # =====================================================================
   # Data collection (identical to standard RLinf)
   # =====================================================================
   env:
     data_collection:
       enabled: True
       save_dir: ${runner.logger.log_path}/collected_data
       export_format: "pickle"
       only_success: False

Step 3: Register the MI Potential Model Type
==============================================

Register the ``mi_potential`` type in RLinf's reward model registry.

Copy ``docs/rlinf_integration/rlinf_workers_reward/mi_potential_reward_model.py``
to ``rlinf/workers/reward/mi_potential_reward_model.py``, then add to the reward
worker registry:

.. code-block:: python

   # rlinf/workers/reward/__init__.py or the corresponding registry file

   from rlinf.workers.reward.mi_potential_reward_model import MIPotentialRewardModel

   REWARD_MODEL_REGISTRY["mi_potential"] = MIPotentialRewardModel

Online Reward Formula
=====================

For each frame transition :math:`(o_t, a_t, r^\text{env}_t, o_{t+1}, \text{done}_t)`:

**Step 1**: Compute state potentials

.. math::

   V_t &= V_\phi(o_t, g) \\
   V_{t+1} &= V_\phi(o_{t+1}, g)

**Step 2**: Compute and clip potential difference

.. math::

   \Delta_t^\text{raw} &= \gamma \cdot V_{t+1} - V_t \\
   \Delta_t^\text{clip} &= \text{clamp}(\Delta_t^\text{raw}, -\delta_\text{clip}, +\delta_\text{clip})

**Step 3**: Confidence gating

.. math::

   c_t &= \min(c_t^\text{cached}, c_{t+1}^\text{next}) \quad \text{(or geometric mean)} \\
   \Delta_t^\text{gated} &=
   \begin{cases}
   c_t \cdot \Delta_t^\text{clip} & \text{if confidence\_gate = true} \\
   \Delta_t^\text{clip} & \text{otherwise}
   \end{cases}

**Step 4**: Final reward

.. math::

   r_t^\text{final} =
   w_\text{env} \cdot r_t^\text{env}
   +
   w_\text{mi} \cdot \lambda_\text{scale} \cdot \Delta_t^\text{gated}

where :math:`w_\text{env}` = ``env_reward_weight``, :math:`w_\text{mi}` = ``reward_weight``,
:math:`\lambda_\text{scale}` = ``scale``, :math:`\delta_\text{clip}` = ``delta_clip``.

Potential cache lifecycle:

.. code-block:: text

   Episode start:
     env.reset() → o_0 → V_phi(o_0, g) → [cached]

   Each step t:
     step(a_t) → o_{t+1} → V_phi(o_{t+1}, g)
     delta_t = γ·V(o_{t+1}) - V(o_t)   ← V(o_t) from cache
     reward = env_reward + mi_delta
     V(o_t) ← V(o_{t+1})               ← update cache

   Episode end (done/reset):
     Clear cache → new episode re-initializes

.. warning::

   The cached potential from the previous episode **must never** be used in
   the next episode. ``PotentialShapingState.clear(env_ids)`` is called
   before every reset to enforce this.

Ablation Configurations
========================

The following ablations can be achieved via YAML or CLI overrides — no Python code changes needed.

**A. Standard RLPD (Baseline)**

.. code-block:: yaml

   reward.potential_shaping.enabled: false

**B. Environment sparse reward only**

.. code-block:: yaml

   reward.use_reward_model: false

**C. Standard binary reward model (ResNet)**

.. code-block:: yaml

   reward.model.model_type: "resnet"
   reward.reward_output_type: "success_probability"

**D. MI potential shaping only (Phase A)**

.. code-block:: yaml

   reward.potential_shaping.enabled: true
   reward.reward_weight: 1.0
   algorithm.mi_guided_replay.enabled: false

**E. MI-guided replay only (Phase B)**

.. code-block:: yaml

   reward.potential_shaping.enabled: true
   reward.reward_weight: 0.0            # shaping weight = 0
   algorithm.mi_guided_replay.enabled: true
   # Note: even with reward_weight=0, potential differences are still computed
   # to label each transition's mi_delta

**F. Full method (MI shaping + MI-guided replay)**

.. code-block:: yaml

   reward.potential_shaping.enabled: true
   reward.reward_weight: 1.0
   algorithm.mi_guided_replay.enabled: true
   reward.potential_shaping.confidence_gate: true

FAQ
===

**Q: Training fails with `No module named 'mi_reward'`**

A: Ensure the MI repository is ``pip install -e``'d into the training node's Python environment.

**Q: Potential differences are always zero**

A: Check:
  1. ``potential_shaping.enabled: true``
  2. ``warmup_steps`` is not too large
  3. ``delta_clip`` is not set to 0
  4. The checkpoint is correctly loaded (check RewardWorker logs)

**Q: Potential differences contain NaN**

A: Check:
  1. ``input_dim`` in ``metadata.json`` matches the actual feature dimension
  2. Image preprocessing is consistent with training
  3. Model loaded successfully (confidence should be 1, valid should be True)

**Q: Robot behavior degrades when MI potential drops**

A: This is partly expected — negative deltas signal "regression", and the SAC Critic
learns to avoid those actions. If it keeps dropping without recovery:
  1. Lower ``reward_weight`` (e.g. 0.5)
  2. Increase ``delta_clip`` negative-side symmetry
  3. Verify the environment reward is not overwhelmed by MI shaping

**Q: Can MI potential directly determine task success?**

A: No. The potential is a progress signal, not a binary success/failure classifier.
Success should always be determined by:
  - Environment ground truth (simulation)
  - Human labels or sensors (real robot)
  - The environment's own binary reward

Safety Notes
============

.. warning::

   1. **MI potential must not trigger reset or episode termination.** Related config is hard-coded to ``false``.
   2. **MI potential must not bypass** workspace limits, force limits, collision detection, emergency stop, human intervention, or episode timeout.
   3. **On model timeout / NaN / invalid output**: log a rate-limited warning, set MI shaping to zero for that step, preserve the environment reward, and do not interrupt the control loop.
   4. **Potential-difference clipping** (``delta_clip``) must be kept in a reasonable range (default 1.0).
   5. **The model is loaded once** (at worker init) and **never** updated online.
   6. **Inference always runs in eval mode** (``torch.inference_mode()``).
   7. On the first Franka run, **set ``reward_weight: 0.0``** first, confirm logs are normal, then gradually increase.

Data Collection
================

The MI potential model does **not** require additional real-robot data collection.
The data collection flow is identical to standard Franka RLPD:

.. code-block:: yaml

   env:
     data_collection:
       enabled: True
       save_dir: ${runner.logger.log_path}/collected_data
       export_format: "pickle"
       only_success: False

Collected episodes can be used for:
  - Demo Buffer (RLPD's 50% demonstration data)
  - Evaluating the MI potential model's real-robot generalization
  - Future reward-model fine-tuning

**Important**: The MI potential model's training data comes from world-model-generated
futures, not real-robot collection. Real-robot collection is only used for RLPD
policy training and validation.
